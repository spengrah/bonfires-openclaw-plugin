# Wave 9 Plan — PM14/PM15 Plan-Phase Close (2026-03-07)

## Objective
Close `/dev` Phase 2 planning for ingestion expansion by finalizing PM14 (PDF lane) and PM15 (linked-content lane) decisions, verification plan, and back-pressure policy before implementation.

## Scope
- PM14: PDF ingestion routing for profile-scanned workspace content.
- PM15: Explicit per-link confirmed ingestion for user-provided links.

## Goals
1. Keep plugin thin and deterministic.
2. Reuse shared ingestion core routing semantics across triggers.
3. Preserve fail-open behavior and per-item fault isolation.

## Non-goals (current phase)
1. DOCX or other binary parsers.
2. Autonomous external discovery (feature 1c).
3. Browser/JS-rendered extraction path.

## Confirmed product decisions
1. PM14 stays Active (Phase A).
2. Duplicate semantics: `success:true` + `message:"duplicate"` => successful no-op.
3. PM15 uses explicit tool + explicit ask flow.
4. PM15 confirmation granularity is per-link (not batch-only).
5. Domain trust policy for PM15: allow user-provided public links, with transport safety guards.
6. Content support target for PM15: PDFs + common text files + HTML extracted readable text.

## Architecture decisions
1. Introduce shared ingestion core (classification/routing/result normalization).
2. Cron ingestion and link-ingestion tool are trigger adapters into the same core.
3. Route rules:
   - `.pdf` / `application/pdf` => `/ingest_pdf`
   - text-like / extracted HTML => `/ingest_content`

## Back-pressure / blocker policy
Block implementation progress if any condition is hit:
1. SSRF guard behavior for linked fetches is ambiguous or bypassable.
2. HTML extraction path is non-deterministic / untestable.
3. Bonfires endpoint contract mismatch discovered in integration checks.
4. Shared-core refactor introduces regressions in PM4/PM6 behavior.

Escalation and decision owner:
- Escalate immediately to Spencer for product/security tradeoff decisions.
- Do not silently downgrade safety controls.

1. Verification Quality
